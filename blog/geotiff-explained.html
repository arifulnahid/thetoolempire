What Is a GeoTIFF and What's Actually Inside It

You have opened a .tif file. Maybe it shows satellite imagery, a DEM, or a land-cover classification. QGIS renders it correctly but you have no idea where the coordinates come from or what all those tags mean. This is the explainer nobody writes.

A TIFF file with geographic memory

A regular TIFF is just a container for pixel data — rows and columns of numbers. A GeoTIFF is the same container with an extra suitcase of geographic metadata stuffed into it. That suitcase includes where the image is on Earth, what coordinate system it uses, how big each pixel is in real-world units, and often what software created it.

The geo metadata lives in special TIFF tags that are invisible to a regular image viewer. ImageMagick convert will render a GeoTIFF just fine, but it will not tell you the projection or bounding box. For that you need GDAL, QGIS, or a browser tool that reads those tags.

The GeoTIFF specification was first published in 1994 by Niles Ritter and Mike Ruth, standardized in 2019 as OGC standard 19-008r4. It is the dominant format for raster GIS data worldwide.

The three layers of metadata

GeoTIFF metadata falls into three groups, each stored differently in the file:

  1. Standard TIFF tags — image width, height, bits per sample, compression method, photometric interpretation. These are defined in the TIFF spec and present in every TIFF file.
  2. GeoTIFF geo keys — coordinate reference system, geotransform, model type. Stored in a custom tag called GeoKeyDirectory (tag ID 34736) and optionally GeoAsciiParams (34737) and GeoDoubleParams (34736).
  3. GDAL metadata — software name, creation date, no-data value, projection overrides, and freeform XML. Stored in tags like GDAL_METADATA, GDAL_NODATA, ImageDescription, and Software. These are unofficial but universally supported.

When QGIS says it cannot find a projection, it means layer 2 is missing or the file uses a custom CRS that is not in the built-in EPSG database. The image can still be displayed, but it will not line up with other layers.

The geotransform — six numbers that place every pixel on Earth

The geotransform is the most important piece of geo metadata. It is a 6-element array — usually written as GT — that defines the affine relationship between pixel coordinates and geographic coordinates:

/* Map pixel (col, row) → geographic (X, Y) */ X = GT[0] + col × GT[1] + row × GT[2] Y = GT[3] + col × GT[4] + row × GT[5] /* For a north-up, non-rotated image */ GT[0] = top-left X (longitude / easting) GT[1] = pixel width in X direction (positive, W→E) GT[2] = rotation coefficient (0 for north-up) GT[3] = top-left Y (latitude / northing) GT[4] = rotation coefficient (0 for north-up) GT[5] = pixel height in Y direction (negative, N→S)

For a north-up image (the common case), GT[2] and GT[4] are zero. GT[5] is negative because rows increase downward while latitude increases upward. A 10-metre resolution DEM covering part of Europe might have:

GT = [5.0, 0.0000899, 0, 53.0, 0, -0.0000899] /* Origin at 5°E, 53°N. ~10m pixels in geographic degrees */

To find the bottom-right corner of a 5000×5000 image: X = 5.0 + 5000 × 0.0000899 = 5.4495, Y = 53.0 + 5000 × (−0.0000899) = 52.5505.

The CRS — which Earth are we measuring?

Coordinates alone are meaningless without knowing which coordinate reference system they use. The geotransform gives you numbers; the CRS tells you what those numbers represent.

GeoTIFF stores the CRS in the GeoKeyDirectory tag using keys defined in the GeoTIFF spec:

Key IDNameWhat it stores
1024GTModelTypeGeoKey1=projected, 2=geographic, 3=geocentric
2048GeographicTypeGeoKeyGeographic CRS EPSG code (e.g. 4326 for WGS 84)
3072ProjectedCSTypeGeoKeyProjected CRS EPSG code (e.g. 32632 for UTM 32N)
2049GeogCitationGeoKeyHuman-readable CRS name string
3073PCSCitationGeoKeyHuman-readable projected CRS name

An EPSG code is a numeric identifier maintained by the IOGP (International Association of Oil & Gas Producers). EPSG:4326 is WGS 84 geographic (the GPS standard). EPSG:3857 is Web Mercator (used by Google Maps and OpenStreetMap tiles). EPSG:32632 is WGS 84 / UTM zone 32N. There are over 9,000 EPSG codes covering every national grid, datum, and projection in common use.

The GeoAsciiParams tag stores a vertical bar-delimited string of CRS name components. It is the human-readable backup when the EPSG codes are absent or custom.

Band types and data formats

Unlike JPEG or PNG which store red/green/blue in one interleaved byte array, GeoTIFF stores multiple bands that can have completely different numeric types:

SampleFormatBitsPerSampleTypical use
Unsigned integer8Natural colour imagery, land-use classification
Unsigned integer16Raw digital elevation model values (0–65535 m)
Signed integer16Elevation with negative values, −32768 to 32767
IEEE float32Processed DEM, temperature, precipitation, NDVI
IEEE float64High-precision scientific rasters

A 4-band Sentinel-2 image has bands 1–4 as uint16 (raw sensor counts 0–10000 representing surface reflectance × 10000). A SRTM DEM has a single band of int16. A GDAL-reprojected DEM often uses float32. The GDAL_NODATA tag tells GIS software which numeric value means "no measurement here" — common values are −9999, −32768, 0, or the float NaN equivalent.

Compression methods — what each one costs you

GeoTIFF supports multiple compression codecs, each with different trade-offs between file size, encoding speed, decoding speed, and lossy/lossless behaviour:

CompressionLossy?Best forNotes
NoneNoProcessing pipelines, fastest readLargest files — 3× to 10× bigger than LZW
LZWNoGeneral-purpose scientific dataGood for integer rasters, moderate ratio
Deflate/ZIPNoFloat rasters, DEMsBetter ratio than LZW for floating point data
ZSTDNoCloud storage, fast decodeBetter ratio than Deflate, faster decode
JPEGYesRGB imagery, photo-qualityNot suitable for DEMs or classified data — introduces artefacts at tile edges
WebPOptionalCOG imagery for web deliverySmaller than JPEG at equivalent quality, lossy or lossless
PackBitsNoOlder software compatibilityMinimal compression, fast decode

Rule of thumb: use Deflate or ZSTD for DEMs and classification rasters, JPEG or WebP for natural colour imagery where small artefacts are acceptable. Never use JPEG for data products — it will corrupt your values in ways that are invisible to the eye but destructive for analysis.

Tiled vs stripped layout

A TIFF can store its pixel rows in one of two internal layouts:

  • Stripped — rows are grouped into horizontal strips of N rows each. Reading any single pixel requires loading the full strip. Fast for sequential row access, terrible for random spatial access.
  • Tiled — the image is divided into rectangular tiles (256×256 or 512×512 are common). Reading a small area only requires loading the relevant tiles. Essential for large files and cloud storage.

Always use tiled layout for files larger than a few hundred megabytes. GDAL's default is stripped; force tiling with -co TILED=YES. A tile size of 512 is a good default for most use cases. Smaller tiles mean more tiles to read for large areas; larger tiles waste bandwidth for small requests.

Cloud Optimised GeoTIFF (COG) — the format built for streaming

A COG is a GeoTIFF whose internal structure has been optimised for HTTP range requests. This lets a client read only the bytes it needs from a file hosted on S3, GCS, or any HTTP server — without downloading the whole file.

Three requirements make a GeoTIFF cloud-optimised:

  1. Tiled layout — as described above. Without tiling, a single tile request still fetches a full strip.
  2. Internal overviews — reduced-resolution copies of the full image, stored as additional IFDs (Image File Directories) inside the same file. A zoom-level 8 web map request reads a small overview; a full-resolution analysis reads the base IFD. GDAL creates overviews with gdaladdo -r average yourfile.tif 2 4 8 16.
  3. IFD ordering — overviews come before the full-resolution data in the file, so a client can find them with a single range request to the file header instead of jumping around.

COG is now the preferred format for publishing raster data on the web. AWS Open Data, Microsoft Planetary Computer, and Google Earth Engine all distribute data as COGs. Tools like rio-cogeo and GDAL's /vsicurl/ virtual file system make it easy to read COGs remotely without downloading them.

BigTIFF — when your data outgrows 4 GB

The classic TIFF format uses 32-bit offsets to locate data within the file, capping the maximum file size at 4 GB. BigTIFF uses 64-bit offsets, removing this limit. A single Sentinel-2 scene at full 10m resolution is already ~800 MB for one band; a full tile stack easily exceeds 4 GB.

BigTIFF files use magic bytes 43 instead of 42 at offset 2 in the file header (after the byte-order marker). GDAL creates BigTIFF automatically for large outputs, or explicitly with -co BIGTIFF=YES. Most modern GIS software reads BigTIFF transparently. Older software from before ~2008 may not.

Reading GeoTIFF metadata without installing anything

GDAL's gdalinfo is the canonical command-line tool for reading GeoTIFF metadata:

gdalinfo yourfile.tif # Shows: driver, size, CRS (in WKT), geotransform, bounding box, # band count, data type, no-data value, statistics, metadata

In Python, rasterio provides a cleaner API:

import rasterio with rasterio.open('yourfile.tif') as src: print(src.crs) # CRS object with .epsg property print(src.transform) # Affine geotransform print(src.bounds) # BoundingBox(left, bottom, right, top) print(src.count) # Band count print(src.dtypes) # Data type per band print(src.nodata) # No-data value print(src.meta) # Full metadata dict
